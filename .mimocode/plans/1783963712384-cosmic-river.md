# Plan: Wire Up FIFO Batch Tracking & Real P&L COGS

## Problem
The `stock_batches` table exists but is never written to. `compute_fifo_cogs()` exists but is never called. P&L uses a hardcoded `amount * 0.6` for COGS instead of actual purchase costs.

## Goal
When an item is bought at 100 (batch 1) then 105 (batch 2), and sold twice, the FIFO cost of goods sold should be 100 + 105 = 205 (not a hardcoded 60% ratio).

---

## Changes (4 files)

### 1. `src/server/purchase_routes.rs` — Create batches on purchase

**`create_direct_purchase` (line ~503, after stock_balances update):**
- After inserting the purchase and updating stock_balances, INSERT into `stock_batches` with:
  - `batch_no`: derived from `p.purchase_no`
  - `item_id`, `warehouse_id`, `source_type = 'PURCHASE'`, `source_id = purchase id`
  - `quantity_original = form.quantity`, `quantity_remaining = form.quantity`
  - `unit_cost = form.unit_cost`
  - `received_date = form.purchase_date`
- Update `purchases.batch_id` to the new batch id.

**`create_goods_receipt` (line ~412, inside the item loop after stock update):**
- For each received item, look up `purchase_order_items.unit_price` using `item.po_item_id` to get the unit cost.
- INSERT into `stock_batches` with the same pattern as above, `source_type = 'GRN'`.

**`update_direct_purchase` (line ~532):**
- If the purchase exists, update the corresponding `stock_batches` row (match by `source_type = 'PURCHASE' AND source_id = purchase_id`) to reflect new quantity and cost.

**`delete_direct_purchase` (line ~522):**
- Delete the corresponding `stock_batches` row.

### 2. `src/server/inventory_routes.rs` — FIFO consumption on OUT movements

**`create_stock_movement` (line ~773, after stock_balances update):**

For `movement_type == "OUT"`:
- Query oldest batches with `quantity_remaining > 0` ordered by `received_date ASC, id ASC`.
- Consume from each batch until the requested quantity is fulfilled:
  - Deduct `min(batch.quantity_remaining, remaining_needed)` from `quantity_remaining`.
  - Build a `BatchConsumption` record.
- If total available in batches < requested quantity, still process (allow negative stock as before), but only consume what's available.
- After consuming, compute FIFO unit cost = total consumed cost / total consumed quantity.
- UPDATE the stock_movement row's `unit_cost` to the computed FIFO cost.
- Set `batch_id` on the stock_movement to the last consumed batch id.

For `movement_type == "IN"`:
- INSERT a new `stock_batches` row with the movement details.

### 3. `src/server/report_routes.rs` — Real COGS in P&L

**`report_profit_loss` (line 199-207):**
Replace the hardcoded `* 0.6` with actual FIFO COGS:

```rust
// Calculate COGS from consumed batches
let cogs: f64 = db.query_row(
    "SELECT COALESCE(SUM((quantity_original - quantity_remaining) * unit_cost), 0)
     FROM stock_batches WHERE quantity_remaining < quantity_original",
    [], |r| r.get(0),
).unwrap_or(0.0);
```

This uses the batch consumption data: for each batch, the consumed quantity is `(quantity_original - quantity_remaining)`, and the cost of that consumption is `consumed_qty * unit_cost`.

### 4. `src/calculations/stock.rs` — Add a helper to compute COGS from DB

Add a convenience function `compute_fifo_cogs_from_batches()` that wraps the SQL query pattern, keeping the pure calculation logic in the calculations module alongside the existing `compute_fifo_cogs()`.
